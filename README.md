# Ubuntu_check_script  
## 우분투 취약점 점검 스크립트

본 저장소는 **Ubuntu Linux 환경을 대상으로 주요 정보통신기반시설 기술적 취약점 분석·평가 기준에 따라 취약점을 점검하는 자동화 스크립트 프로젝트**입니다.  
단순히 “취약 / 양호”를 판별하는 수준을 넘어, **점검 기준 해석 → 조건 분기 → 오탐 최소화 → 결과 분류(취약 / 양호 / 검토)**까지 고려하여 작성하는 것을 목표로 합니다.

---

## 프로젝트 목적

- Ubuntu 서버 환경에서 **보안 설정 미흡으로 발생할 수 있는 취약점**을 체계적으로 점검
- **공식 가이드 문서의 모호한 표현을 스크립트 로직으로 해석·구현**
- 자동화 도구가 놓치기 쉬운 **환경별 예외 케이스와 검토 항목 분리**
- 보안 컨설팅 관점에서 **“왜 취약한지”를 설명할 수 있는 결과 출력**

---

## 기준 문서

- 주요정보통신기반시설 기술적 취약점 분석·평가 상세 가이드
- Unix / Linux 보안 점검 항목 (U-01 ~ U-70)

> 가이드 문서의 문장을 그대로 코드화하지 않고,  
> 실제 운영 환경에서 의미 있는 판별 조건으로 재해석하여 구현합니다.

---

## 구현 범위

- **운영체제**
  - Ubuntu 22.04 기준 테스트
- **점검 대상**
  - 계정 및 인증 관리
  - 파일 및 디렉터리 권한
  - 서비스 설정 (FTP, DNS, NFS, Mail, NIS 등)
  - 네트워크 서비스 노출 여부
  - 시스템 초기화 스크립트 및 환경 변수

---

## 스크립트 설계 원칙

### 1. 가이드 우선, 환경 고려
- “설정이 없다 = 무조건 취약”이 아닌  
  **서비스 미설치 / 비활성화 / 설정 미사용 여부를 구분**
- inetd / xinetd / systemd 환경 차이를 모두 고려

### 2. 결과 분류 기준 명확화
- **취약 (Vulnerable)**  
  즉시 조치가 필요한 보안 설정 오류
- **양호 (Good)**  
  가이드 기준 충족
- **검토 (Review)**  
  - 운영 목적상 허용될 수 있으나 위험 요소가 존재하는 경우  
  - 자동 판별이 어려운 항목

### 3. 오탐 최소화
- 주석 제거 후 설정 분석
- 대소문자, 공백, 설정 표현 방식 차이 고려
- 설정의 **존재 여부**와 **실제 의미**를 분리하여 판단

---

## 결과 출력 예시

```text
U-25 취약: /usr/bin/example 파일이 실행 가능 파일이며 other write 권한이 설정됨
U-36 검토: r-command 서비스 패키지는 존재하나 서비스 비활성 상태
U-43 양호: NIS 관련 서비스 비활성 확인
