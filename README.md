# Ubuntu_check_script  
## 우분투 취약점 점검 자동화 스크립트

이 저장소는 **Ubuntu Linux 환경을 기준으로 주요정보통신기반시설 기술적 취약점 분석·평가 가이드(KISA)를 참고해 작성한 취약점 점검 스크립트 프로젝트**이다.  

단순히 취약 여부만 출력하는 도구가 아니라,  
가이드 문서의 점검 기준을 실제 시스템 설정과 비교하면서  
**어디까지 자동으로 판단할 수 있는지**,  
그리고 **어떤 부분은 사람이 직접 확인해야 하는지**를 구분하는 것을 목표로 했다.

실무 운영에 바로 사용하는 도구라기보다는,  
**보안 점검 기준을 기술적으로 해석하고 자동화 로직으로 옮겨보는 과정 자체에 의미를 둔 프로젝트**이다.

---

## 프로젝트 목적

- Ubuntu 서버 환경에서 **보안 설정 미흡으로 발생할 수 있는 취약 요소 점검**
- 가이드 문서에 명시된 기준 중  
  **표현이 모호하거나 해석이 필요한 항목을 조건 분기 로직으로 구현**
- 자동 판단이 어려운 항목은 **검토(Review) 대상으로 분리**
- 점검 결과를 확인할 수 있도록  
  **관련 파일 경로, 디렉터리 위치, 설정 값 등을 함께 출력**

---

## 기준 문서

- 주요정보통신기반시설 기술적 취약점 분석·평가 상세 가이드
- Unix / Linux 보안 점검 항목

---

## 구현 범위

### 운영체제
- Ubuntu 22.04 LTS 기준으로 테스트

### 점검 대상
- 계정 및 인증 설정
- 파일 및 디렉터리 권한
- 서비스 설정  
  (FTP, DNS, NFS, Mail, NIS 등)
- 네트워크 서비스 노출 여부
- 시스템 초기화 스크립트 및 환경 변수 설정

---

## 구현 내용 요약

- 가이드 기준을 단순히 코드로 옮기기보다는  
  **서비스 미설치 / 비활성화 / 설정 미사용 여부를 구분해 판단**
- 자동 판별이 가능한 항목과  
  **환경이나 운영 목적에 따라 해석이 필요한 항목을 분리**
- “설정이 없다 = 무조건 취약”으로 처리하지 않도록 로직 구성

이 프로젝트를 통해  
**기준 문서를 읽고 → 기술적으로 해석하고 → 자동화 가능한 형태로 구현하는 흐름**을 직접 경험했다.

---

## 스크립트 설계 기준

### 1. 가이드 기준 우선
- 가이드 문서에 명시된 점검 의도를 우선적으로 반영
- 설정 파일의 존재 여부뿐 아니라  
  **실제 설정 값의 의미를 기준으로 판단**

### 2. 결과 분류
- **취약 (Vulnerable)**  
  즉시 조치가 필요하다고 판단되는 설정 상태

- **검토 (Review)**  
  - 운영 목적에 따라 허용될 수 있으나 보안 리스크가 존재하는 경우  
  - 자동화로 단정하기 어려워 관리자의 판단이 필요한 항목

### 3. 오탐 최소화
- 주석 제거 후 설정 분석
- 대소문자, 공백, 설정 방식 차이 고려
- 설정의 **존재 여부**와 **동작 의미**를 구분하여 처리

---

## 결과 출력 예시

```text
U-52 취약: telnet 서비스가 설치되어 있으며 /etc/inetd.conf에 활성화되어 있음
U-56 검토: vsFTP ftpusers 파일에 사용자 계정이 등록되어 있어 운영 목적 확인 필요
```
